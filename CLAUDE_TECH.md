## 산출물 유형별 기본 구조

### A. 시스템 아키텍처 문서

```
1. 시스템 개요
   - 목적 및 범위
   - 주요 이해관계자 및 역할

2. 아키텍처 설계
   - 전체 구성도 (6계층 기준)
   - 컴포넌트별 역할 정의
   - 데이터 흐름도 (Data Flow Diagram)

3. 인터페이스 정의
   - 외부 시스템 연동 목록
   - API 명세 (Endpoint / Method / Payload 형식)
   - 프로토콜 및 데이터 포맷

4. 데이터 모델
   - 엔티티 정의 및 관계 (ERD)
   - 핵심 속성 및 데이터 타입
   - 동기화 주기 및 정합성 기준

5. 비기능 요구사항
   - 성능: 응답시간 / 처리량 (TPS)
   - 가용성: SLA 목표치 (예: 99.9%)
   - 보안: 인증 방식 / 암호화 수준
   - 확장성: Scale-out 전략

6. 기술 스택 선정 근거
   - 후보 기술 비교표 (기준: 성능 / 비용 / 생태계 / 국내 레퍼런스)
   - 최종 선정 및 이유
```

### B. 요구사항 정의서 (PRD)

```
1. 제품 개요
   - 목표 (Goal): 1문장
   - 배경 및 필요성
   - 성공 지표 (KPI): 측정 가능한 수치로 정의

2. 사용자 정의
   - 페르소나 (Persona): 역할 / 목표 / pain point
   - 사용 시나리오 (User Story)
     형식: "나는 [역할]로서, [목적]을 위해 [기능]을 사용한다"

3. 기능 요구사항
   - 기능 목록 (Feature List)
   - 우선순위: MoSCoW (Must / Should / Could / Won't)
   - 각 기능별: 입력 / 처리 / 출력 / 완료 조건

4. 비기능 요구사항
   - 성능 / 보안 / 가용성 / 유지보수성

5. 제약 조건
   - 기술적 제약
   - 예산 / 일정 제약
   - 규제 / 표준 준수 요건

6. 미결 사항 (Open Issues)
   - 확인 필요 항목 목록
```

### C. 기술 비교 분석

```
1. 비교 목적 및 기준 정의
2. 후보 기술/제품 목록
3. 비교 매트릭스 (표 형식)
   - 기준별 점수 (5점 척도 + 근거)
4. 종합 평가 및 권고안
5. 도입 시 고려사항
```

### D. WBS (기술 프로젝트용)

```
Phase 1: 분석 및 설계
  1.1 현황 분석 (As-Is)
  1.2 요구사항 정의
  1.3 아키텍처 설계
  1.4 기술 스택 확정

Phase 2: 개발
  2.1 데이터 수집 계층 구축
  2.2 데이터 통합 계층 구축
  2.3 DT 엔진 개발
  2.4 서비스/응용 개발
  2.5 UI 개발

Phase 3: 통합 및 검증
  3.1 단위 테스트
  3.2 통합 테스트
  3.3 성능 테스트
  3.4 사용자 수락 테스트 (UAT)

Phase 4: 배포 및 운영
  4.1 운영 환경 구성
  4.2 데이터 마이그레이션
  4.3 운영 매뉴얼 작성
  4.4 교육 및 이관

각 Task 필수 포함 항목:
  - 담당 역할
  - 기간 (시작일 ~ 종료일)
  - 완료 조건 (Gate): 검증 가능한 기준
  - 산출물
  - 선행 조건 (Dependency)
```
